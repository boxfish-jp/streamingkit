# プログラミング配信において、現在の進捗を表示する

# 欲しい情報
- 現在のタスク
- 現在のタスクの現在地(完了率?)
- ゴールが何で、何をめざしているか

現在のタスクと完了率でよいのかも。

# たたき台に表示するもの
- 親タスク名(例:プログラミング配信において、現在の進捗を表示する)
- 現在終了しているタスクのタスク全体における割合
- 現在取り組んでいる小タスク名

# 欲しい情報の取得方法

## orgファイルから
- 現在THINKING, DEVELOPING, TEST, BUILDINGになっているタスクが小タスク
- その小タスクが所属している親タスク
- その親タスクがもっているタスク数と、現在終了しているタスク数

## orgファイルの見つけ方
/devディレクトリ以下のプロジェクトを検索していき、todo.orgファイルを見つけたら最終更新時間を確認して、最も最新のものを使用する。

## 実装方法
CLIアプリを作って、監視する。  
最終更新時間とデータ内容を保存しておいて、最終更新時間が変わったらデータを更新する。  
変更された行を特定して、そこから現在のタスクと完了率を計算する。  

パーサーは自作する。  
UIの更新指示をCLIアプリでWebsocketを使って出す。
アプリ内の状態管理はファイルに保存を現在はしない。したくなった時に実装する。

ファイル読み取りはchokidarを使い、flockを使ってロックする。  

## パーサの要件定義(これは実装前で実装後に改善された)

### タスク進捗抽出・ツリー生成処理のデータ仕様

本処理は、テキストベースのタスクリストの「旧データ」と「新データ」を比較し、更新箇所（直近の進捗）および該当タスクのツリー構造を抽出してJSON形式で出力するものです。

#### 1. 入力データとステータスタグの定義

入力データとして、以下の「Org-mode風のプレーンテキスト」を**2つ（変更前の旧データと、変更後の新データ）**受け取ります。
アスタリスク（`*`）の数で階層（親タスク・子タスク）を表現し、その後に「ステータスタグ」と「タスク名」が続きます。

**【ステータスタグの種類とルール】**
入力データに含まれるステータスタグは、以下のグループに分類されます。

* **未着手:** `TODO`、または**タグの記述がない状態**
* **作業中タグ:** `THINKING`, `DEVELOPING`, `TEST`, `BUILDING`
* **完了タグ:** `DONE`, `CANCELED`

**※タグなしの扱いに関する特記事項:**

* ステータスタグが何も書かれていないタスクは、`TODO` と同様に「未着手」として扱われます。
* 子タスクを持つタスク（親タスク）に関しては、ステータスタグを記述しなくても、明示的にタグを付与しても、どちらでもよい仕様となっています。

**【入力データのイメージ】**

```text
▼ 旧データ
* プロジェクトA                  ← (タグなしの親タスク)
** TODO タスクC                  ← (タグありの親タスク)
*** DEVELOPING 放置されているタスクB
*** TODO サブタスクD
*** タグなしのサブタスクE         ← (未着手として扱われる)

▼ 新データ
* プロジェクトA
** TODO タスクC
*** DEVELOPING 放置されているタスクB
*** DEVELOPING サブタスクD       ← (ステータスが更新された！)
*** タグなしのサブタスクE

```

#### 2. 「現在作業中のタスク」の定義

本仕様において、「現在作業中のタスク」とは、単に作業中タグがついているものではなく、以下の**2つの条件を両方とも満たすタスク**と厳密に定義します。

1. 新データにおけるタグが**作業中タグ**（`THINKING`, `DEVELOPING`, `TEST`, `BUILDING`）であること。
2. 旧データと新データで**タグが異なる**こと（今回ステータスが更新されたタスクであること）。
※タグなし（未着手）から作業中タグへ変更された場合も含まれます。

※旧データから継続して作業中タグのままとなっているタスク（放置タスクなど）は、ここには含まれません。

#### 3. 出力データ（最終結果）

新旧データの比較・集計を経て、最終的に以下の2つの要素（`recentProgress` と `treeData`）を持つJSONデータを出力します。

##### A. 最近の進捗トピック (`recentProgress`)

今回のデータ更新で発生した「差分」のリストです。

* **`activeTasks`**: 上記で定義した「現在作業中のタスク」のリスト。
* **`doneTasks`**: 今回の更新で、新しく**完了タグ**（`DONE`, `CANCELED`）に変わったタスクのリスト。
* **`ancestors`（先祖情報）**: そのタスクがどの親タスクに属しているかを示すルートからのリスト（例: `["プロジェクトA", "タスクC"]`）。

##### B. 抽出されたツリーデータ (`treeData`)

「現在作業中のタスク」を起点とし、それに関係するタスク（先祖、兄弟、子孫）のみを残して木構造に再構成したデータです。

* **ハイライトフラグ (`isActive`):**
`isActive: true` となっているノードは、`activeTasks` に登録されているタスクであることを示します。兄弟タスクの中に「作業中タグのまま放置されているタスク（`isActive: false`）」と「今回新しく作業中になったタスク（`isActive: true`）」が入り混じっているツリーにおいても、このフラグによって「現在まさに作業中であるタスク」を正確に見分けることができます。
* **全体の正確な進捗データ (`stats`):**
ツリーの抽出処理によって除外されたタスクが存在する場合でも、各親ノードの `stats` には、入力データ全体から算出された「本来の全タスク数 (`total`)」と「完了タグの数 (`doneCount`)」が正確に集計・保持されています（※タグなしタスクも `total` にカウントされます）。

**【treeData型定義】**

```typescript
// タスクの取り得るステータス
// ※入力テキスト上で「タグなし」のものは "TODO" として扱われます
export type TaskStatus = 
  | "TODO"
  | "THINKING"
  | "DEVELOPING"
  | "TEST"
  | "BUILDING"
  | "DONE"
  | "CANCELED"
  | "NEW"; // 旧データに存在しなかった（新規追加された）場合の oldStatus 用

// -------------------------------------------------------------
// 1. 最近の進捗トピック用データ (recentProgress)
// -------------------------------------------------------------

// ステータスが変化した個別のタスク情報
export interface RecentTaskChange {
  id: string;
  title: string;
  status: TaskStatus;      // 新データでのステータス
  oldStatus: TaskStatus;   // 旧データでのステータス
  ancestors: string[];     // ルートからの親タスク名のリスト (例: ["プロジェクトA", "タスクC"])
}

// 最近の進捗をまとめたオブジェクト
export interface RecentProgress {
  activeTasks: RecentTaskChange[]; // 今回の更新で新たに「作業中」になったタスク
  doneTasks: RecentTaskChange[];   // 今回の更新で新たに「完了」になったタスク
}

// -------------------------------------------------------------
// 2. 抽出されたツリーデータ (treeData)
// -------------------------------------------------------------

// タスク配下の正確な進捗状況（ツリーから除外されたタスクも計算に含む）
export interface TaskStats {
  total: number;
  doneCount: number;
}

// ツリー構造の1ノード
export interface TreeNode {
  id: string;
  title: string;
  status: TaskStatus;
  isActive: boolean;    // 更新された「現在作業中のタスク」であるか（ハイライト用）
  stats: TaskStats;     // フルデータから計算された正確な進捗
  children: TreeNode[]; // 子ノードの配列
}

// -------------------------------------------------------------
// 3. 最終的な出力データ全体
// -------------------------------------------------------------

// バックエンド処理から出力され、UIへ渡されるJSONの全体像
export interface TaskUpdateResponse {
  recentProgress: RecentProgress;
  treeData: TreeNode[];
}
```

## メッセージ実装
### 処理の流れ
cliがファイルの変更の監視をする。  

### フロントエンド
メッセージには
- 更新情報を送信するメッセージ(前回のファイル内容と今回のファイル内容)。  
- 描画のオンオフの指示。
フロントエンドがパースをし、UIに流し込む。
フロントエンドはあんまり状態を持ってほしくない。

### パーサー
パーサーはフロントエンドに表示したいものに合わせて結果を変える必要があるので、
ドメイン的にはフロントエンドに近い。

### 描画タイミング
ファイルが変更されたとき、来場したとき、コメントが来たときに合わせる。  
フロントエンドからは遠い、オーケストレーターに近い？
オーケストレーターは指示だけしてほしい。  

### CLIがやること
chokidarはファイルの変更イベントを監視してくれるだけ。
ファイルの読み込みと前回の状態の保存をCLIがやる必要がある。
ファイルが更新されたとき更新情報をフロントエンドに送る。
